## 作成にあたって考えたこと

### async/await(非同期処理)を用いたAPIアクセス、情報取得処理

GitHubと猫画像取得APIにて実践してみました。

ロード中に他の要素の稼働が止まらないということが現代においては必須であることを学びました。
  
### useStateによるAPI情報取得状態、ユーザーのフォームへの入力状態の保持

API情報のみならず、ユーザーの入力保持にも利用を模索したのですが、これに関連して疑問（勘違い）とそれに伴う気付きがありました。

#### **useStateで監視・保持していたユーザー入力の文字列を、エンコード、明示的なエスケープ等せずに``alert()``の引数に入れていたことについて**

XSSなどの脆弱性につながるのではないか？と考え、一度該当部分を削除しました。

一方で、

1. ``alert()``関数は引数を文字列として解釈すること

2. React,Next.jsが変数代入時に自動エスケープを行っていること

を学び、仮にこれらのセーフガードがないとしても、

URLを引数に取っているわけではないので、反射XSSにすらならず、

実行者が自身のマシンでJavaScriptコードを実行するにとどまる「セルフXSS」となるに過ぎない可能性が高い、ことを学習できました。

### useEffectによる状況変化に基づくアクションのトリガー

GitHubと、猫画像の取得関数をトリガー。

### localStorageプロパティによる状態保存で、再読み込みをまたがった状態記憶

送信(モック)ボタン押下で、入力された文字列に応じてalertの内容を分岐、

送信機能は未実装の旨と猫画像再読み込み、猫画像のidにジャンプ(スクロール)する、

という簡単なギミックを想定していました。

当初、画面全体をリロードする仕様にしていたことから、

ステートレスなWebではリロード後にuseStateに寄る状態保持がそもそもできない、という初歩的な学習ができました。

全体リロード構成では、localStorageプロパティにより、ボタンが押された時のリロードかどうか、のフラグを管理する必要があったため、意味があった部分と考えています。

### iframeに自身の公開プレイリストを埋め込み、Spotifyとの連携を意識（アカウント側のプレイリスト名変更、曲追加削除をリニアにサイトに反映する）

これが一番フロントエンド開発での楽しみを感じた部分です。

iframe埋めこみだけなら、WordPressやはてなブログなどのブログでも可能かと思いますが、

一から(AI支援があるとはいえ)作った自分のサイトが、Spotifyという超大型サービスと連携する(そして、情報取得だけでなくその場での再生も可能)、というのはとても感慨深いものでした。

フロントエンド開発の醍醐味(？)を少しだけ味わうことができ、今後のささやかな学習モチベーションに繋がりました。

## 今後の課題

- ``alert()``のダイアログがブラウザ依存であり、とにかくダサい。ユーザーの操作を制限するため、ユーザー体験的にもNGだと感じる。
 - React(TS)でページ自体を変化させ、ユーザーの操作を邪魔しないUI設計が必要。

- 単一ページで、シンプルと言えば聞こえはいいが、拡張の余地がある。
 - 複数ページの連携、データのやり取りを学んでいく。
